2. The Site

The site runs on AWS — Amazon's cloud. The major pieces:

  • Frontend — React app (what visitors see)
  • API — Node.js Express server in a container
  • Database — PostgreSQL with 28,000+ roses
  • File storage — AWS S3 for uploads and images
  • Domain & SSL — Auto-renewed by AWS

You don't touch any of this directly. The admin dashboard does most of what you need.

3. Daily Operations

You do nothing daily. The site runs itself. Alerts arrive by email when something needs attention.

Alarm Email Subject Means Action
rosebase-app-down Site is unreachable Check the site, see Troubleshooting
rosebase-alb-5xx-spike App is throwing errors Check logs, may need restart
rosebase-enrichment-no-start Tonight's enrichment didn't run Often self-resolves on next run
rosebase-job-failure-spike Multiple nightly jobs failed Contact developer if persistent
rosebase-playwright-crash Browser-based scraper crashed Often transient; alert if 3+ in an hour
rosebase-billing-200-warning Bill above $200 Review costs
rosebase-billing-250-action Bill above $250 Investigate immediately

No email = everything is fine.

4. The Nightly Pipeline

Automated jobs run on a schedule. Times in Eastern Time.

Nightly (every day)

02:30  Nursery Audit              Health-checks 1,000+ nurseries
03:00  Enrichment                 Fills missing rose data via OpenAI
03:00  Fast-Lane Refresh          Quick re-enrichment of trending roses
03:00  Session Cleanup            Purges expired user sessions
03:00  Fragrance Consensus        Aggregates fragrance votes into canonical labels
03:30  Confidence Accumulation    Adds confidence to candidate signals based on age
04:00  Catalog Scrape             Updates prices from 115 nursery sites
04:00  Coverage Snapshot          Captures field-completeness metrics
04:30  Cover Audit                AI verifies cover photos
05:01  Deduplication              Merges duplicate rose entries
05:15  Degraded Escalation        Escalates persistently-failing scrapers
05:30  Progress Snapshot          Saves daily metrics

Nightly window finishes by ~6:30 AM ET. Don't deploy code between 2:00 AM and 6:30 AM ET — it interferes with the jobs.

Weekly

Sun 07:00   Variety Photo Audit          AI verifies all variety-level photos
Sun 08:00   Stale Nursery Auto-Retire    Retires nurseries broken 60+ days (capped at 50/run)

Monthly

1st @ 03:00   Photo Sweep              Sources photos for last month's new roses
15th @ 02:00  HathiTrust Ingestion     Imports rose names from HathiTrust historical archives

Quarterly

Jan/Apr/Jul/Oct 1st @ 05:00   Patent Application Monitor   Tracks new rose patent filings

Annual

Jan 1 @ 05:00               Annual Patent Refresh         Refreshes all patent records

Admin-triggered (no fixed schedule)

These workers are registered with pg-boss but do not auto-fire. Trigger them manually from /admin?section=nightly-ops:

  • weekly-dead-url-check — Re-checks links flagged as broken
  • weekly-color-accuracy-audit — AI re-verifies rose color labels
  • weekly-genealogy-audit-verify — Verifies parent/child rose links
  • weekly-photo-workers-mass — Bulk re-runs photo enrichment
  • weekly-inactive-nursery-retry — Re-attempts dormant nurseries
  • weekly-full-pipeline — Runs the entire nightly pipeline on-demand
  • weekly-run-health-audit — Generates the health-audit report
  • monthly-nursery-discovery — Discovers new nurseries via Serper
  • monthly-hybridizer-monitoring — Refreshes hybridizer profiles
  • candidate-triage — Processes pending rose-candidate queue
  • immediate-rose-enrichment — One-off enrichment for a single rose ID

weekly-event-discovery is listed in BOTH schedulesToClear (with comment "[SERPER SUSPENDED]") AND in boss.schedule() (fires monthly on the 1st @ 06:00 ET). The schedule() call wins — it currently runs monthly and costs ~255 Serper queries per run. If this is intentional, ignore. If suspended was the intent, remove the boss.schedule() call. 5. The Admin Dashboard

Sign in at https://rosesroses.com/admin with an admin account. The left sidebar lists every section. Below are the most common ones.

Orders & Fulfillment

Where: /admin?section=orders

RosesRoses operates as a concierge — customers pay you, you place the order with the nursery on their behalf, the nursery ships directly to the customer. This is a manual workflow you do for each order.

The status lifecycle:

Status What it means What you do
Awaiting Payment Customer started checkout but hasn't paid Nothing — wait for Stripe
Payment Received Customer paid you via Stripe Place the order with the nursery, then mark "Ordered from Nursery"
Ordered from Nursery You've placed the order Wait for the nursery to ship
Shipped Nursery has shipped to customer Wait for delivery
Delivered Customer received the roses Done
Cancelled Order cancelled at any stage Refund via Stripe if already paid

How to fulfill an order (typical flow):

  1. You get an email: "New order received — Order #1234"
  2. Open /admin?section=orders. The new order shows Payment Received.
  3. Click into the order. You'll see: - Customer's name, email, shipping address - The rose(s) they ordered, which nursery to buy from, the quantity, the price you owe the nursery - A Nursery Product URL link — opens the nursery's product page directly
  4. Place the order with the nursery — go to the nursery's website, buy the rose, ship it to the customer's address (not yours).
  5. Back in the admin order page, paste the nursery's order number into Nursery Order Number, change status to Ordered from Nursery, and save. The customer gets an email update automatically.
  6. When the nursery ships, paste the tracking number into the order, change status to Shipped. Customer gets another email update.
  7. When the customer confirms delivery (or after a reasonable wait), mark Delivered. Done.

How payment works:

  • Customer pays the full amount (rose price + shipping + tax + your service fee) to you via Stripe.
  • The money lands in your Stripe account, then transfers to your bank account on Stripe's payout schedule (typically 2 business days).
  • You pay the nursery out of pocket when you place the order. The customer's payment plus your service fee covers the cost + your margin.
  • If you ever need to refund, do it from the Stripe dashboard (dashboard.stripe.com) — the order status here doesn't auto-refund.

Adjusting your service fee:

Where: Admin > Commercial > Sales tab > Service fee settings

You control your cut from this screen — toggle between a percentage (e.g., 15%) or a flat dollar amount (e.g., $4 per order), and change it any time. The value applies to all new orders going forward; existing orders keep whatever fee was active at checkout.

Adding or Managing Nurseries

Where: /admin?section=commercial&tab=cn-nurseries

Lists every nursery in the database. Click one to:

  • Edit name, website, country, state
  • Set scraper config (which CSS selectors to use)
  • Set tier (Gold / Silver / Bronze) — affects auto-approval threshold
  • Mark as active/inactive or block

To add a new nursery: 1. Click Add Nursery 2. Fill in name, country, website 3. Save — the next nightly catalog scrape will start scraping it

Approving Nursery Submissions

Where: /admin?section=nurseries&tab=submissions

When users submit nurseries via the public submission form, they land here as Pending. Review:

  • Nursery name, website, country
  • Submitter name (so you can credit/contact them)

Click Approve to add it to the active nursery list, or Reject with a reason.

Moderation Queue

Where: /admin?section=moderation-queue

Centralized queue for everything user-submitted that needs review:

  • Comments — already public, but you can delete abusive ones
  • Fact proposals — corrections users want to make to rose data
  • Directory proposals — new hybridizers, distributors, growers
  • Photo uploads — pending photos for roses
  • Genealogy proposals — submitted parent/offspring relationships

Each row has Approve and Reject buttons. Approved entries become live; rejected ones are removed.

Candidate Queue (AI-Discovered Roses)

Where: /admin?section=candidate-queue

The nightly scrapers discover new rose names from nursery websites. Names that aren't already in the database land here.

  • Auto-approved candidates from Gold/Silver-tier nurseries skip this and go straight in
  • Pending candidates need your review — confirm "yes this is a real rose" or reject

The Confidence Score column tells you how sure the system is. Anything ≥50 is usually safe to approve.

Coverage Dashboard

Where: /admin?section=coverage

A single view of pipeline health: how many roses, how many nurseries are active vs broken, listings count, growth trends. Useful as a daily glance.

Manually Triggering a Nightly Job

Where: /admin?section=nightly-ops

Run any nightly job on demand — useful when: - A scrape failed and you want to re-run it - You added a nursery and want it scraped now (don't wait until 4 AM) - You're testing pipeline changes

Click the job name and confirm. Watch progress in the same panel.

Suspending an Abusive User

Where: /admin?section=user-support

Search for a user by email, click into their profile, Suspend with a reason. They're immediately logged out and can't sign in until you unsuspend.

6. Common Tasks (Infrastructure)

Restart the App

Sometimes a restart fixes weird issues:

aws ecs update-service --cluster rosebase --service rosebase-api --force-new-deployment --region us-east-1

Takes ~2 minutes. No downtime.

Deploy a Code Change

  1. Push to the main branch on GitHub
  2. GitHub auto-builds and deploys (~10-15 min)
  3. Don't deploy between 2:00–6:30 AM ET

Update an API Key (Stripe, OpenAI, etc.)

  1. AWS Console → search Secrets Manager
  2. Find the secret (e.g., rosebase/openai-api-key)
  3. Click → Retrieve secret value → Edit
  4. Paste new value, save
  5. Restart the app (above)

View Logs

  • Terminal: aws logs tail /ecs/rosebase --follow --region us-east-1
  • AWS Console: CloudWatch → Log groups → /ecs/rosebase

8. Troubleshooting

Site won't load / errors

  1. Visit https://rosesroses.com/api/healthz. Should show {"status":"ok"}.
  2. AWS Console → ECS → rosebase cluster → events tab. Look for crash messages.
  3. CloudWatch Logs → /ecs/rosebase — search for error near the time.
  4. Restart the app (Section 6).
  5. If still broken, contact a developer.

Database is slow

  • AWS Console → RDS → rosebase → Monitoring tab
  • If CPU consistently >80%, scale up: Modify → instance class → db.t4g.large
  • 5-10 min downtime during the change

A nightly job didn't run

  • You'll get an alert email automatically
  • Check CloudWatch logs at the expected time (02:30-05:30 ET)
  • Most jobs auto-recover the next night thanks to checkpoint logic
  • Contact developer if a job fails 3+ nights in a row

9. Backups & Security

Backups happen automatically. The database is backed up daily, kept for 30 days. Files in S3 are versioned, so deleted ones can be recovered for 30 days. If you ever need to actually restore from a backup, call your developer — it's a rare event and not something to do casually.

Security is already configured. HTTPS everywhere, a firewall blocking common attacks, API keys locked in AWS Secrets Manager, and an audit log of every AWS action. If you suspect a security incident (someone stole a password, unusual activity, etc.), call your developer immediately — don't try to fix it yourself.

10. Glossary

For non-technical readers — what these terms mean:

Term What It Is
AWS Amazon Web Services. The cloud platform hosting your site.
Region A geographic location of AWS data centers. Yours is us-east-1 (N. Virginia).
Console The AWS web interface — like a control panel for your infrastructure.
ECS Elastic Container Service. The thing that runs your app.
Fargate The "serverless" mode for ECS — AWS manages the underlying servers.
Container A package of your code + everything it needs to run.
Task A single running instance of your container. You have 1 task running.
Service The configuration that keeps your task running (auto-restarts on crash).
RDS Relational Database Service. The PostgreSQL database.
S3 Simple Storage Service. Where files and images are stored.
ALB Application Load Balancer. Routes incoming traffic to your app.
VPC Virtual Private Cloud. Your isolated network in AWS.
NAT Gateway Lets your private servers reach the internet (e.g., to scrape sites).
Route 53 AWS's DNS service — converts rosesroses.com to an IP address.
ACM AWS Certificate Manager. Provides the SSL certificate (the padlock icon).
WAF Web Application Firewall. Blocks attacks and abusive traffic.
Secrets Manager A safe place for API keys and passwords (encrypted, audited).
CloudWatch AWS's monitoring system. Stores logs, sends alerts.
CloudTrail Audit log of every action taken in AWS.
CI/CD Continuous Integration / Continuous Deployment. Auto-deploys your code.
GitHub Actions The system that runs CI/CD when you push code.
DNS The phone book of the internet. Maps domain names to IP addresses.
Sitemap A file listing every page on your site so search engines can find them.
API Application Programming Interface. The "backend" the frontend talks to.
pg-boss A library that runs scheduled jobs (the nightly pipeline).
Stripe The company that processes credit card payments.
